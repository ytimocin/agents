## Concept Model

Helm has three primary nouns:

| Term | What it is |
|------|------------|
| **Chart** | A versioned package of Kubernetes templates + default values. The unit Helm installs. |
| **Release** | A specific install of a chart into a cluster, with a name and a sequence of revisions. |
| **Repository** | A place charts come from — either a classic HTTP repo (`index.yaml` + `.tgz`) or an OCI registry (`oci://...`). |

Helm 3+ is **client-only** — there is no in-cluster Tiller anymore. The CLI uses your kubeconfig directly; release state is stored as Secrets (default driver) in the release's namespace under `sh.helm.release.v1.<name>.v<rev>`.

Full docs: https://helm.sh/docs/glossary/ · Architecture intro: https://helm.sh/docs/intro/

## Chart File Structure

```
mychart/
├── Chart.yaml            # required — chart metadata
├── Chart.lock            # generated by `helm dependency update`
├── values.yaml           # default values
├── values.schema.json    # optional JSON Schema validating .Values
├── README.md
├── LICENSE
├── .helmignore           # patterns excluded from packaging (gitignore syntax)
├── charts/               # vendored subchart .tgz files
├── crds/                 # CRDs installed before templates render (NOT templated)
└── templates/
    ├── _helpers.tpl      # named templates (underscore-prefixed = no manifest output)
    ├── deployment.yaml
    ├── service.yaml
    ├── NOTES.txt         # printed after install/upgrade
    └── tests/
        └── test-connection.yaml
```

Helm reserves the names `charts/`, `crds/`, `templates/`, `Chart.yaml`, `Chart.lock`, `values.yaml`, `values.schema.json`, `README.md`, `LICENSE`, `.helmignore`. Underscore-prefixed files (`_helpers.tpl`) are partials — they're loaded but produce no manifest. `NOTES.txt` is special — its rendered content prints after install.

Full docs: https://helm.sh/docs/topics/charts/#the-chart-file-structure

## Chart.yaml

```yaml
apiVersion: v2                # v2 = Helm 3+; v1 is legacy with separate requirements.yaml
name: mychart                 # lowercase + digits + hyphens; no underscores/uppercase
version: 1.4.2                # SemVer 2 — chart version
appVersion: "8.0.1"           # SemVer of the underlying app (string; quote it)
type: application             # application | library
kubeVersion: ">=1.27.0-0"     # SemVer constraint on cluster version
description: ...
home: https://example.com
sources: [https://github.com/...]
keywords: [database]
maintainers:
  - name: jane
    email: jane@example.com
    url: https://...
icon: https://.../icon.png
deprecated: false
annotations: {}
dependencies:
  - name: postgresql
    version: "13.x"
    repository: "oci://registry.example.com/charts"
    condition: postgresql.enabled
    tags: [database]
    alias: pg
    import-values:
      - child: service
        parent: postgresService
```

**Chart types:**
- `application` (default) — installable; templates render to manifests.
- `library` — **not installable**; provides reusable named templates only. Set `type: library` and reference from another chart's `dependencies`.

**Versioning:** Charts and `appVersion` must follow SemVer 2. Don't put `+build` in chart version if you also use it in a Kubernetes label — labels disallow `+`; replace with `_` (`{{ .Chart.Version | replace "+" "_" }}`).

Full docs: https://helm.sh/docs/topics/charts/#the-chartyaml-file · Library charts: https://helm.sh/docs/topics/library_charts/

## Templating

Helm renders templates with **Go's `text/template`** plus the **Sprig** function library (with `env` and `expandenv` removed for safety) and a few Helm-specific functions. The output is YAML.

### Built-in objects

| Object | Key fields |
|--------|------------|
| `.Release` | `Name`, `Namespace`, `IsInstall`, `IsUpgrade`, `Revision`, `Service` (always `"Helm"`) |
| `.Chart` | All of `Chart.yaml` (`Name`, `Version`, `AppVersion`, `Type`, …) |
| `.Values` | Merged values (chart defaults + `-f` files + `--set`) |
| `.Files` | `Get`, `GetBytes`, `Glob`, `Lines`, `AsConfig`, `AsSecrets` — read non-template files in the chart |
| `.Capabilities` | `KubeVersion`, `APIVersions.Has`, `HelmVersion` — gate templates by cluster/Helm version |
| `.Template` | `Name` (current template path), `BasePath` (templates dir) |
| `.Subcharts.<name>` | Subchart's own scope (its `.Values`, `.Chart`, …) |

### Control structures, scoping, and pipelines

```gotemplate
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
{{- end }}

{{- with .Values.podAnnotations }}        {{/* `.` is rebound to .Values.podAnnotations inside */}}
annotations:
  {{- toYaml . | nindent 4 }}
{{- end }}

{{- range .Values.containers }}
- name: {{ .name | quote }}
  image: {{ printf "%s:%s" .image .tag | quote }}
{{- end }}
```

`{{- ` and ` -}}` trim adjacent whitespace (including newlines) — essential for keeping YAML valid.

### Helm-specific functions worth knowing

| Function | Purpose |
|----------|---------|
| `include "name" .` | Render named template, return as string (pipeable — unlike `template`). |
| `tpl "{{ .Values.foo }}" .` | Evaluate a string as a template at render time. |
| `required "msg" .Values.foo` | Fail rendering with a clear error if value is empty/missing. |
| `lookup "v1" "Secret" "ns" "name"` | Read live cluster state at render time (returns dict; empty if not found). |
| `toYaml`, `toJson`, `toToml` | Marshal a value to YAML/JSON/TOML — usually piped through `nindent`/`indent`. |
| `default "x" .Values.y` | Provide a fallback. |

**`include` vs `template`:** `template` is a Go-template action and **cannot be piped**. `include` is Helm's pipeable wrapper — always prefer `include` so you can `| nindent N`.

### Named templates and `_helpers.tpl`

```gotemplate
{{/* templates/_helpers.tpl */}}
{{- define "mychart.fullname" -}}
{{- if .Values.fullnameOverride -}}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" -}}
{{- else -}}
{{- printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" -}}
{{- end -}}
{{- end -}}

{{- define "mychart.labels" -}}
app.kubernetes.io/name: {{ include "mychart.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version | replace "+" "_" }}
{{- end -}}
```

Files starting with `_` are loaded but produce no manifest. Convention is to put all named templates in `templates/_helpers.tpl`.

### `.Files` for embedded data

```gotemplate
data:
  config.yaml: |
{{ .Files.Get "config/app.yaml" | indent 4 }}
```

`Files.Glob` returns a map; `AsConfig` produces a YAML map of basename→content (perfect for `ConfigMap.data`); `AsSecrets` does the same with base64 values for `Secret.data`. **`.Files` cannot reach outside the chart directory.**

Full docs: https://helm.sh/docs/chart_template_guide/ · Built-in objects: https://helm.sh/docs/chart_template_guide/builtin_objects/ · Function list: https://helm.sh/docs/chart_template_guide/function_list/ · Named templates: https://helm.sh/docs/chart_template_guide/named_templates/ · Control structures: https://helm.sh/docs/chart_template_guide/control_structures/ · Variables: https://helm.sh/docs/chart_template_guide/variables/ · Accessing files: https://helm.sh/docs/chart_template_guide/accessing_files/

## Values

### Precedence (lowest → highest)

1. Chart's own `values.yaml`.
2. Parent chart's `values.yaml` (when this chart is a subchart).
3. User-supplied `-f` files, **in the order they appear on the command line** (later files win).
4. Inline `--set`, `--set-string`, `--set-file`, `--set-json` flags (later flags win).

To remove a key from defaults at install time, set it to `null`: `--set livenessProbe.httpGet=null`.

### `--set` semantics

| Flag | Treats input as | Notable behavior |
|------|------------------|------------------|
| `--set k=v` | Inferred type | Bare numbers, bools, and `null` parsed as their type. Commas separate keys; backslash-escape with `\,`. Dots traverse keys; `\.` keeps a dot in the key name. |
| `--set-string k=v` | Always string | Use when a numeric-looking value (e.g., a tag like `1.0`) must stay a string. |
| `--set-file k=path` | Reads file content | Whole file becomes the string value — handy for shoving a cert or script into a value. |
| `--set-json k=<json>` | Raw JSON | Lets you set lists/objects without YAML files (`--set-json 'replicas=[1,2,3]'`). |

### Values in templates

```gotemplate
{{ .Values.image.repository }}
{{ .Values.podAnnotations | toYaml | nindent 4 }}
```

### Subchart values

A parent reaches into a subchart with the subchart's name: `.Values.postgresql.auth.database`. Subcharts cannot read parent or sibling values directly. **Globals** declared under `global:` are merged into every chart's `.Values.global` and are the supported way to share values across charts.

### `values.schema.json`

JSON Schema validating the merged `.Values` during `install`, `upgrade`, `lint`, `template`. Catches typos and bad types before render. Disable with `--skip-schema-validation` if you must.

Full docs: https://helm.sh/docs/chart_template_guide/values_files/ · Subchart values: https://helm.sh/docs/topics/charts/#scope-dependencies-and-values · Schema: https://helm.sh/docs/topics/charts/#schema-files

## Dependencies & Subcharts

Define under `Chart.yaml > dependencies`. Then:

```bash
helm dependency update    # fetches into charts/, writes Chart.lock
helm dependency build     # rebuilds charts/ from Chart.lock (CI / reproducible)
helm dependency list      # show deps + status
```

Dependency fields:
- `condition` — single boolean path; if false the subchart is omitted (e.g., `postgresql.enabled`).
- `tags` — alternative to `condition`; a tag is on if any dep using it sets it true.
- `alias` — install the same chart twice under different names (`primary`, `replica`).
- `import-values` — promote child values into a parent path (`child: service` → `parent: postgresService`) or via the child's `exports:` block.

**CRDs in subcharts** are loaded from each chart's `crds/` directory; Helm does not template them and only installs them once. Helm does **not** upgrade or delete CRDs after the initial install (intentional — see best-practices link below).

Full docs: https://helm.sh/docs/topics/charts/#chart-dependencies · Best practices for CRDs: https://helm.sh/docs/chart_best_practices/custom_resource_definitions/

## Hooks

Mark any manifest as a hook with the `helm.sh/hook` annotation. Hooks are rendered as part of the release but executed at lifecycle points; resources Helm creates from hooks are **not tracked as part of the release** unless you opt in with `helm.sh/resource-policy: keep`.

| Hook | Fires |
|------|-------|
| `pre-install`, `post-install` | install |
| `pre-upgrade`, `post-upgrade` | upgrade |
| `pre-rollback`, `post-rollback` | rollback |
| `pre-delete`, `post-delete` | uninstall |
| `test` | `helm test <release>` |

Order knob: `helm.sh/hook-weight` (string-typed integer, default `"0"`). Sorts ascending across all hooks at the same phase, then by kind, then by name.

Cleanup knob: `helm.sh/hook-delete-policy` (comma-separated list):
- `before-hook-creation` (default) — delete the previous run before launching a new one.
- `hook-succeeded` — delete after exit 0.
- `hook-failed` — delete after non-zero exit.

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: "{{ .Release.Name }}-migrate"
  annotations:
    "helm.sh/hook": pre-upgrade,pre-install
    "helm.sh/hook-weight": "-5"
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: migrate
        image: myapp:{{ .Values.image.tag }}
        command: [/bin/sh, -c, "./migrate.sh"]
```

Hooks block: Jobs/Pods run synchronously; Helm waits for completion before continuing the phase.

Full docs: https://helm.sh/docs/topics/charts_hooks/

## Tests

Tests are Pods (or Jobs) under `templates/` (convention: `templates/tests/`) annotated `helm.sh/hook: test`. Container exit code 0 = pass.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: "{{ .Release.Name }}-test-connection"
  annotations:
    "helm.sh/hook": test
spec:
  restartPolicy: Never
  containers:
  - name: curl
    image: curlimages/curl:8.5.0
    args: ["-fsSL", "http://{{ include "mychart.fullname" . }}/healthz"]
```

Run with `helm test <release>`. Add `--logs` to stream pod logs; `--filter name=<n>` to run a single test. Tests are not run by `helm install` unless you also pass `--wait` and the chart relies on a separate testing pipeline — typically you run `helm test` after `helm install`.

Full docs: https://helm.sh/docs/topics/chart_tests/

## Repositories vs OCI Registries

### Classic chart repository (HTTP)

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo bitnami/postgres
helm pull bitnami/postgresql --version 13.4.0 --untar
```

A repo is just an HTTP server hosting `index.yaml` (a catalog) plus packaged `.tgz` files. `helm repo index` regenerates the catalog when you publish.

### OCI registry (preferred for new work)

Charts are pushed as OCI artifacts to any OCI-compliant registry (ECR, GHCR, ACR, Artifactory, Harbor, GAR, Docker Hub).

```bash
helm registry login -u <user> registry.example.com
helm package mychart                          # produces mychart-1.4.2.tgz
helm push mychart-1.4.2.tgz oci://registry.example.com/charts
helm install web oci://registry.example.com/charts/mychart --version 1.4.2
helm pull oci://registry.example.com/charts/mychart --version 1.4.2 --untar
```

**Gotchas:**
- The `oci://` reference must **not** include the chart's filename or tag — Helm computes them from the package and `--version` flag.
- `helm registry login` host is hostname-only — no scheme, no path.
- You can pin to a digest: `helm install web oci://registry.example.com/charts/mychart@sha256:...`.
- Subchart `dependencies.repository` accepts `oci://...`.

Full docs: https://helm.sh/docs/topics/registries/ · Chart repos: https://helm.sh/docs/topics/chart_repository/

## CLI Reference (cheat-sheet)

### Lifecycle

```bash
helm install   <name> <chart> [flags]    # create release
helm upgrade   <name> <chart> [flags]    # update existing release
helm upgrade --install <name> <chart>    # upsert (idempotent install/upgrade)
helm rollback <name> <revision>          # revert to a prior revision
helm uninstall <name>                    # delete a release (use --keep-history to retain)
```

Most-used `install`/`upgrade` flags:

| Flag | Effect |
|------|--------|
| `-f, --values <file>` | Layer a values file (repeatable; later wins). |
| `--set k=v`, `--set-string`, `--set-file`, `--set-json` | Inline overrides. |
| `-n, --namespace <ns>` | Target namespace. |
| `--create-namespace` | Create the namespace if missing (install only). |
| `--version <constraint>` | Pin chart version (semver). |
| `--dry-run` | Render + send to apiserver in dry-run mode; doesn't persist. |
| `--debug` | Verbose logging including rendered manifests. |
| `--wait` | Block until all resources are Ready (or timeout). |
| `--wait-for-jobs` | With `--wait`, also wait for hook Jobs. |
| `--timeout <dur>` | How long `--wait` waits (default `5m0s`). |
| `--atomic` | If install/upgrade fails, automatically rollback. Implies `--wait`. |
| `--cleanup-on-fail` | Delete newly created resources on failure (no rollback). |
| `--post-renderer <bin>` | Pipe rendered manifests through an external program. |
| `--include-crds` / `--skip-crds` | Toggle CRD installation. |
| `--no-hooks` | Skip hooks for this command. |
| `--reset-values` / `--reuse-values` / `--reset-then-reuse-values` | Upgrade-only: how previous values interact with new flags. |

### Inspect

```bash
helm list [-A] [-n ns] [--all] [--filter REGEX]
helm status   <name>                     # current state of a release
helm history  <name>                     # revisions
helm get all|manifest|values|hooks|notes|metadata <name> [--revision N]
helm show    chart|values|readme|crds|all <chart>
```

### Render & validate locally

```bash
helm template <name> <chart> -f values.yaml [--set ...] [--debug]   # local render only
helm lint     <chart> [-f values.yaml]                              # static checks
helm dependency update <chart>                                      # sync charts/ + lockfile
helm package  <chart>                                               # produce .tgz
```

`helm template` is offline (does not contact the API server) **except** when the chart calls the `lookup` function. Use it in CI.

### Repos & registries

```bash
helm repo add|update|list|remove|index
helm search hub <q>          # Artifact Hub (cross-repo)
helm search repo <q>         # local repo cache
helm registry login|logout
helm push   <chart.tgz>  oci://<host>/<path>
helm pull   <chart>     [--version X] [--untar]
```

### Tests, plugins, env

```bash
helm test <release> [--logs] [--filter name=<n>]
helm plugin install|list|update|uninstall|verify
helm env                     # print HELM_* environment variables
helm version                 # client (and server, but server is gone in v3)
```

## Release Storage & State

State is stored as Secrets (default) in the release's namespace, one Secret per revision: `sh.helm.release.v1.<release>.v<revision>`. Override via `HELM_DRIVER=configmap|sql|memory`.

`--history-max` (upgrade flag, default 10) caps revision count — older revisions are pruned.

Atomic install/upgrade: `--atomic` rolls back on failure; `--cleanup-on-fail` removes newly created resources without flipping back to the prior revision.

Full docs: https://helm.sh/docs/topics/advanced/

## Best Practices (high-leverage)

### Naming & conventions
- Chart name: lowercase + digits + `-` only.
- Chart version: SemVer 2; replace `+` with `_` when used in K8s labels.
- 2-space YAML indent; never tabs.
- Don't hardcode `metadata.namespace` in templates — let `--namespace` set it. This keeps charts portable to Flux/ArgoCD/Spinnaker.

### Standard labels (apply to every resource)

```yaml
metadata:
  labels:
    app.kubernetes.io/name:       {{ include "mychart.name" . }}
    app.kubernetes.io/instance:   {{ .Release.Name }}
    app.kubernetes.io/version:    {{ .Chart.AppVersion | quote }}
    app.kubernetes.io/managed-by: {{ .Release.Service }}
    app.kubernetes.io/component:  api          # user-defined
    app.kubernetes.io/part-of:    myplatform   # user-defined
    helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version | replace "+" "_" }}
```

### Values structure
- Lowercase camelCase keys. Document each key in `values.yaml` with a comment.
- Keep the tree shallow; avoid deep nesting that subcharts then have to re-traverse.
- Provide sensible defaults that work with `helm install --generate-name` and no `--set`.
- Validate with `values.schema.json`; fail fast.

### Templates
- Quote strings, **don't** quote integers.
- Use `include` (not `template`) when you need to pipe (`| nindent`, `| sha256sum`).
- Wrap required values with `required "msg"` so missing config fails clearly.
- Restart pods on config change with a checksum annotation:

  ```yaml
  spec:
    template:
      metadata:
        annotations:
          checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
  ```
- Avoid generating random data without seeding — `randAlphaNum 16` regenerates on every upgrade and triggers churn. Prefer `lookup` to reuse the prior secret, or pass a value.

### CRDs
- Place plain (non-templated) YAML in `crds/`. Helm installs them once before templates render.
- Helm does **not** upgrade or delete CRDs. For lifecycle management, install CRDs separately or use a tool like `cert-manager`'s approach.

Full docs: https://helm.sh/docs/chart_best_practices/ · Conventions: https://helm.sh/docs/chart_best_practices/conventions/ · Values: https://helm.sh/docs/chart_best_practices/values/ · Templates: https://helm.sh/docs/chart_best_practices/templates/ · Labels: https://helm.sh/docs/chart_best_practices/labels/ · CRDs: https://helm.sh/docs/chart_best_practices/custom_resource_definitions/ · RBAC: https://helm.sh/docs/chart_best_practices/rbac/ · Pods: https://helm.sh/docs/chart_best_practices/pods/ · Dependencies: https://helm.sh/docs/chart_best_practices/dependencies/ · Tips & tricks: https://helm.sh/docs/howto/charts_tips_and_tricks/

## Provenance & Signing

Helm supports OpenPGP signing of chart packages.

```bash
helm package mychart --sign --key 'My Signing Key' --keyring ~/.gnupg/secring.gpg
# produces mychart-1.4.2.tgz AND mychart-1.4.2.tgz.prov
helm verify  mychart-1.4.2.tgz                           # offline verify
helm install web mychart-1.4.2.tgz --verify              # verify + install
```

The `.prov` file contains: the chart's `Chart.yaml`, a SHA-256 of the `.tgz`, and a clearsigned PGP signature. GnuPG v2 users must export to a legacy keyring format (`gpg --export-secret-keys > secring.gpg`).

For OCI artifacts, the **helm-sigstore** plugin enables Sigstore (cosign-style) signing.

Full docs: https://helm.sh/docs/topics/provenance/

## Plugins

Plugins extend the CLI. Install with:

```bash
helm plugin install https://github.com/databus23/helm-diff
helm plugin list
helm plugin update diff
helm plugin uninstall diff
```

Each plugin is a directory containing a `plugin.yaml` (name, version, usage, `platformCommand`, `platformHooks`). Located under `$HELM_PLUGINS` (`helm env | grep HELM_PLUGINS`). Helm exposes `HELM_PLUGIN_DIR`, `HELM_PLUGIN_NAME`, `HELM_NAMESPACE`, `HELM_KUBECONTEXT` etc. to the plugin process.

**Downloader plugins** register with `downloaders:` in `plugin.yaml` to handle custom URL schemes (e.g., `s3://`, `git://`).

### Common community plugins (all installed via `helm plugin install <git-url>`)

| Plugin | Purpose |
|--------|---------|
| `helm-diff` | Show what `upgrade` would change before applying. |
| `helm-secrets` | Encrypt/decrypt values files via SOPS. |
| `helm-s3` | Use S3 buckets as a chart repository. |
| `helm-git` | Install charts straight from a Git URL. |
| `helm-unittest` | Pure-template unit tests (no cluster needed). |

Full docs: https://helm.sh/docs/topics/plugins/

## Common Workflows

### Scaffold a new chart

```bash
helm create mychart            # generates a working "hello world" Deployment + Service
cd mychart
helm lint
helm template demo . --debug
```

### Idempotent install (CI-friendly)

```bash
helm upgrade --install web ./mychart \
  -n prod --create-namespace \
  -f values.yaml -f values-prod.yaml \
  --atomic --timeout 5m \
  --version 1.4.2
```

`--install` makes it install on first run and upgrade thereafter. `--atomic` rolls back on failure.

### Render locally for review or GitOps

```bash
helm template web ./mychart -f values-prod.yaml \
  --namespace prod \
  --kube-version 1.30.0 \
  --api-versions apps/v1 \
  > rendered.yaml
```

Useful when feeding the output to `kubectl diff -f rendered.yaml` or to ArgoCD/Flux pre-rendering.

### Diff before upgrade

```bash
helm plugin install https://github.com/databus23/helm-diff
helm diff upgrade web ./mychart -f values-prod.yaml
```

### Rollback

```bash
helm history web
helm rollback web 4         # revision 4
```

### Pull, inspect, modify, and reinstall

```bash
helm pull bitnami/postgresql --version 13.4.0 --untar --untardir ./vendor
# edit ./vendor/postgresql/values.yaml
helm install pg ./vendor/postgresql -n data --create-namespace
```

### Idiomatic `templates/_helpers.tpl`

```gotemplate
{{/* Expand the name of the chart. */}}
{{- define "webapp.name" -}}
{{- default .Chart.Name .Values.nameOverride | trunc 63 | trimSuffix "-" -}}
{{- end -}}

{{/* Fully-qualified release name (≤ 63 chars to fit DNS labels). */}}
{{- define "webapp.fullname" -}}
{{- if .Values.fullnameOverride -}}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" -}}
{{- else -}}
{{- $name := default .Chart.Name .Values.nameOverride -}}
{{- if contains $name .Release.Name -}}
{{- .Release.Name | trunc 63 | trimSuffix "-" -}}
{{- else -}}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" -}}
{{- end -}}
{{- end -}}
{{- end -}}

{{/* Selector labels (immutable — used in spec.selector.matchLabels). */}}
{{- define "webapp.selectorLabels" -}}
app.kubernetes.io/name: {{ include "webapp.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end -}}

{{/* Common labels (selector + version/managed-by/chart). */}}
{{- define "webapp.labels" -}}
helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version | replace "+" "_" }}
{{ include "webapp.selectorLabels" . }}
{{- if .Chart.AppVersion }}

{{- end }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end -}}
```

## Troubleshooting

### `Error: rendered manifests contain a resource that already exists`

The release would create an object that another release (or `kubectl apply`) already owns. Helm 3 refuses to adopt unmanaged resources by default. Either delete the conflicting object or pass `--take-ownership` (Helm 3.14+) when you really do want Helm to take it over.

### `Error: UPGRADE FAILED: another operation (install/upgrade/rollback) is in progress`

A previous command was killed mid-flight, leaving the release in `pending-*` state. Inspect with `helm history <name>` and either:
- `helm rollback <name> <last-good-revision>`, or
- patch the release Secret directly (last resort): `kubectl patch -n <ns> secret sh.helm.release.v1.<name>.v<rev> --type=merge -p '{"metadata":{"labels":{"status":"deployed"}}}'`.

### `Error: timed out waiting for the condition`

`--wait` exhausted `--timeout`. Bump `--timeout` or remove `--wait`. The actual stuck resource is in the cluster — drill in with `kubectl get events -n <ns> --sort-by=.lastTimestamp`.

### Templates render to invalid YAML

- Most common cause: missing `{{- ... -}}` whitespace trim or missing `nindent`.
- `helm template . --debug -f values.yaml | head -200` shows the rendered output with comments.
- `helm lint --strict` catches more issues; `--with-subcharts` validates everything.

### Subchart values aren't being applied

You probably didn't namespace them under the subchart's name. To override `postgresql.auth.password` in the parent's `values.yaml`:

```yaml
postgresql:
  auth:
    password: hunter2
```

Or via flag: `--set postgresql.auth.password=hunter2`. Globals must live under `global:`.

### `lookup` returns empty

`lookup` is a no-op during `helm template` (offline) and `helm install --dry-run`. It only reads cluster state during `helm install`/`upgrade`/`rollback` (or `template --validate`). Plan templates so they degrade gracefully when `lookup` is empty (e.g., generate a new secret if no prior one exists).

### CRDs not upgrading

By design — Helm installs CRDs from `crds/` once and never touches them again. Manage CRD lifecycle outside the chart (separate release, operator, or `kubectl apply -f crds/`).

### `helm install --verify` fails with "no signature found"

The chart wasn't packaged with `--sign`, the `.prov` file isn't next to the `.tgz`, or your local keyring doesn't contain the signer's public key.

Full docs (debugging): https://helm.sh/docs/chart_template_guide/debugging/

## Helm 4 Notes

Helm 4 is the current major line as of 2026. Several docs pages still carry a banner that they have not been fully updated for v4. Behaviors known to differ vs Helm 3:
- Some flags have new defaults (e.g., the `--reset-then-reuse-values` semantic was added in v3 and is the recommended default in v4 — confirm on the live page).
- Helm 4 emphasizes OCI-first distribution; classic chart repos remain supported.
- The Go SDK has had restructuring — see https://helm.sh/docs/sdk/.

When the user is on Helm 4 and the answer hinges on a v3-vs-v4 difference, **WebFetch the relevant page** before committing to specifics rather than relying on this prompt's phrasing.

Full docs: https://helm.sh/docs/overview · Changelog: https://helm.sh/docs/changelog · SDK: https://helm.sh/docs/sdk/

## Answering Style

- Lead with a direct answer. For chart-authoring questions, often the cleanest answer is a small YAML/template diff, not prose.
- Quote exact filenames (`templates/_helpers.tpl`), exact annotations (`helm.sh/hook-weight`), and exact flag spellings.
- When you produce template snippets, keep whitespace trimming (`{{- -}}`) consistent with surrounding YAML.
- Prefer `include` over `template`, `--upgrade --install` over a raw `install`, and `values.yaml` overrides over inline `--set` for anything beyond a one-off.
- If a fact is version-gated (Helm 3 vs Helm 4) or marked "not yet updated for Helm 4" on the source page, hedge and WebFetch before asserting.
- When the user shows a rendering error, your first move is `helm template ... --debug -f <values>` mentally — narrate that diagnosis path.
